# RoseCloud 技术要求说明

## 1. 文档定位

本文档用于整理 `rosecloud` 的技术要求，重点覆盖非功能需求、运行约束、交付要求和工程要求，作为后续架构设计、模块规划、测试验收和运维要求的基础输入。

本文档不讨论具体技术实现，只明确：

- 系统应该达到什么要求
- 哪些要求属于第一阶段必须关注
- 哪些要求属于后续逐步加强

## 2. 非功能需求原则

`rosecloud` 作为平台型系统，技术要求不是补充项，而是基础要求的一部分。

总体原则：

- 第一阶段先满足“可用、可维护、可排障、可演进”
- 不追求一开始做到最强指标
- 先明确约束，再逐步提高标准
- 非功能需求要服务于真实团队协作和长期维护

## 4. 可用性需求

### 4.1 基本可用性

系统需要满足以下基本要求：

- 单体模式可稳定启动
- 微服务模式可稳定启动
- 核心依赖异常时，能够明确暴露故障
- 关键管理功能应具备基本可访问性

### 4.2 运行稳定性

系统应具备：

- 长时间运行的基本稳定性
- 核心服务重启后的可恢复性
- 中间件短时波动下的可恢复性

### 4.3 第一阶段要求

第一阶段重点要求：

- 不追求复杂高可用架构
- 但要保证核心链路稳定
- 不能把故障定位完全依赖人工猜测

第一阶段建议验收基线：

- 单体模式在本地标准环境下可独立启动并完成基础功能验证
- 微服务模式在本地标准环境下可完成网关、认证、系统管理、租户主链路联调
- 核心依赖不可用时，系统应明确报错、降级或拒绝，而不是静默失败

## 5. 性能与容量需求

### 5.1 基本性能要求

系统需要满足：

- 常见管理类接口具备可接受的响应时间
- 登录、鉴权、用户查询、权限查询等基础链路保持稳定
- 后台批处理、消息和任务链路可以支撑第一阶段业务量

### 5.2 容量扩展要求

系统需要具备：

- 从单体模式平滑演进到微服务模式的能力
- 支持后续按服务拆分和扩容
- 支持后续按租户规模扩展数据隔离策略

### 5.3 第一阶段原则

- 第一阶段不做过度性能优化
- 第一阶段优先保证结构可扩展
- 当真实瓶颈出现后，再考虑更重的通信与中间件演进路线

第一阶段建议验收基线：

- 常见管理类接口在常规测试数据量下应保持稳定响应
- 登录、鉴权、菜单加载、权限校验等基础链路应作为重点压测与回归对象
- 任务失败、消息失败、租户初始化失败等后台处理场景应具备明确状态、重试机制和人工介入入口

## 6. 安全需求

### 6.1 认证与授权安全

系统需要满足：

- 用户身份可验证
- 权限边界清晰
- 内外部访问边界明确
- 用户上下文与租户上下文可信
- 密码策略可配置并强制校验（强度、过期、重用约束）
- 失败登录限制与账号锁定
- 用户激活与密码重置支持限时令牌

### 6.2 配置与密钥安全

系统需要满足：

- 敏感信息不直接写入代码仓库
- 密钥、密码、令牌等敏感配置应有明确管理边界
- 环境之间的敏感配置应隔离

### 6.3 审计安全

系统需要具备：

- 登录审计
- 操作审计
- 关键配置变更审计
- 安全策略变更可追踪
- 登录审计记录来源 IP 与客户端环境（浏览器 / 操作系统 / 设备）

### 6.4 增强安全能力

后续需要支持：

- OAuth2
- MFA
- 更细粒度的数据权限
- 更强的访问控制策略
- 会话治理（单用户会话数上限、改密后失效旧令牌）

## 7. 多租户与数据隔离需求

### 7.1 租户隔离要求

系统需要满足：

- 默认关闭多租户时，按单租户模式稳定运行
- 开启多租户后，租户数据边界明确
- 租户隔离失败时，优先拒绝而不是默认放行

### 7.2 数据源与隔离演进要求

系统需要具备：

- 支持后续更强的数据隔离模式演进
- 在演进过程中尽量减少业务层重构范围

### 7.3 租户运营要求

系统需要支持：

- 租户启停
- 租户生命周期管理
- 套餐与配额扩展
- 平台级跨租户操作边界控制

## 8. 可观测性需求

### 8.1 日志需求

系统需要具备：

- 应用日志
- 登录日志
- 操作日志
- 任务日志
- 消息链路日志

### 8.2 指标与监控需求

系统需要具备：

- 基础健康检查
- 核心服务运行状态监控
- 中间件依赖可用性监控
- 关键任务与消息状态监控

### 8.3 链路追踪需求

系统需要支持：

- 请求链路标识
- 网关到下游服务的链路可追踪
- 异步任务和消息链路具备追踪扩展位

## 9. 可维护性需求

### 9.1 模块边界

系统需要满足：

- 模块职责清晰
- 公共能力边界清晰
- 业务代码不直接依赖具体中间件实现

### 9.2 工程可维护性

系统需要满足：

- 适合多人协作开发
- 需求变更后可以局部调整
- 后续增加模块时不会明显破坏现有结构

### 9.3 范围控制

系统需要坚持：

- 不为未来不确定需求提前做过多抽象
- 不把所有能力都做成平台化重模块
- 不把所有增强能力放进第一阶段

## 10. 可测试性需求

系统需要支持以下测试层次：

- 单元测试
- 集成测试
- 关键链路联调测试
- 多租户、多数据源、消息、任务等特殊场景验证

测试要求：

- 核心业务链路应可验证
- 核心平台能力应有基本回归验证方式
- 文档中的关键需求应能逐步映射到验收和测试

## 11. 可部署性需求

### 11.1 本地部署

系统需要满足：

- 本地环境统一
- 基础依赖通过 Docker Compose 编排
- 单体与微服务模式都可在本地运行

### 11.2 环境部署

系统需要支持：

- 本地环境
- 测试环境
- 预发环境
- 生产环境

### 11.3 发布与回滚

系统需要具备：

- 明确的配置切换方式
- 数据库迁移能力
- 发布说明
- 回滚说明

## 12. 文档与协作需求

系统需要具备完整的文档内容维度，包括：

- 需求文档
- 能力边界文档
- 角色与场景文档
- API 文档
- 数据模型文档
- 运维文档
- 安全文档

说明：

- 上述是文档内容维度，不要求当前阶段必须拆成同名独立文件
- 当前阶段可按“需求说明 + 技术要求说明”集中承载，再在后续需要时逐步拆分专题文档

协作要求：

- 文档应能支持立项、设计、开发、联调和运维
- 文档应优先服务团队协作，而不是仅用于展示

## 13. 第一阶段必须关注的非功能需求

第一阶段必须重点关注以下内容：

- 可启动性
- 基本稳定性
- 安全边界清晰
- 多租户默认关闭但边界明确
- 配置与环境可管理
- 日志、审计、健康检查可用
- 文档、测试、迁移、发布占位齐全

## 14. 第二阶段重点增强的非功能需求

第二阶段重点增强：

- 更完整的安全能力
- 更完整的租户运营能力
- 更完整的监控、告警和审计能力
- 更系统的测试与发布质量保障

## 15. 当前阶段结论

现阶段对 `rosecloud` 的非功能需求可以总结为：

- 第一阶段先把系统做成“能跑、能管、能排障、能演进”
- 第二阶段再逐步提高安全、运维、租户、交付质量标准
- 非功能需求必须提前写清，否则后续实现容易失控或返工
