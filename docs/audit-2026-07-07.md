# RoseCloud 代码审核报告

- **日期**：2026-07-07
- **范围**：当前仓库全量源码（275 个 Java 文件，含 `rosecloud-common` / `rosecloud-api` / `rosecloud-starter-*` / `rosecloud-service` / `rosecloud-monolith`）
- **方式**：静态走查 + 关键安全路径逐行核实（过滤器链、JWT 编解码、内部端点、配置）
- **结论**：架构分层与编码规范扎实，基础素养良好；但**安全鉴权设计存在系统性缺口**，当前形态不建议直接用于生产环境。核心问题是「能登录」已做对，「能做什么」基本没做。

| 维度 | 评价 |
| --- | --- |

## 二、安全发现

### 🔴 Critical

**C1 — 全系统缺失方法级鉴权（RBAC），任何有效 JWT 可执行任意特权操作**
- 证据：`grep @PreAuthorize|hasRole|hasAuthority|@EnableMethodSecurity|@Secured|@RolesAllowed` 在全部 Java 源码中**零命中**（仅在 `docs/thingsboard-reference.md` 参考文档中出现）。
- 所有外部 controller（`UserController`、`TenantController`、`RoleController`、`DeptController`、`ConfigController`、`Dict*Controller`、`AuditLogController`、`SessionController`、`MenuController` 等）均无角色/权限校验。
- `SecurityContextFilter.java:58-63` 对无效 token 直接构造空角色的 guest 用户且**不拒绝请求**；网关 `JwtAuthenticationGlobalFilter`、单体 `MonolithJwtFilter` 同样只校验「是否登录」。
- 后果：持任意有效 JWT 即可创建/删除用户、分配角色、开通或停用租户、修改配置字典、查询审计日志。
- 修复：引入 `@EnableMethodSecurity` + `@PreAuthorize("hasRole(...)")`；权限数据来自 `CurrentUser.roles()`；或在 controller 入口统一做角色校验。建议把权限枚举沉淀到 `ErrorCode`/`CommonErrorCode` 同级的权限常量。

### 🟠 High

**H1 — Refresh Token 可被当作 Access Token 直接使用（未校验 token type）**
- 证据：`JwtTokenCodec.parse()`（`JwtTokenCodec.java:65-84`）只校验签名/issuer/过期/type 字段存在性，**不校验 type 必须是 ACCESS**。
- 三处资源访问过滤器均未校验：`JwtAuthenticationGlobalFilter.java:56`、`SecurityContextFilter.java:55`、`MonolithJwtFilter.java:60` 均是 `parse(token)` 后直接放行。
- 全仓**唯一**校验 type 的地方是 `AuthServiceImpl.refresh:85`（`claims.type() != TokenType.REFRESH` → 拒绝）。
- 后果：默认 7 天有效期的 Refresh Token 可直接作为 `Bearer` 调用任意 API，等价于长期 Access Token。
- 修复：在三个过滤器中校验 `claims.type() == TokenType.ACCESS`，否则返回 401。

**H2 — 内部接口返回 `passwordHash` 且完全无鉴权**
- 证据：`InternalUserController.java:17` `@RequestMapping("/internal/users")`，`:26-29` 返回 `UserAuthInfo`（`UserAuthInfo.java:11` 含 `passwordHash` 字段）。
- 同类无认证内部端点：`InternalRevocationController`、`InternalLoginLogController`、`InternalNoticeRecipientController`、`InternalSessionController`（路径均为 `/internal/**`）。
- 风险：集群内任何能直连系统服务端口的 Pod 可无认证拿到任意用户密码哈希。
- 修复：内部接口加服务间认证（mTLS 或内部 token）；对外契约只暴露校验所需字段，避免 `passwordHash` 跨网络传输；叠加网络隔离。

**H3 — 源码提交弱/默认密钥，且关闭 Nacos 鉴权**
- 证据：`docker-compose.yml:7,145` JWT 默认密钥 `rosecloud-dev-secret-please-change-me-0123456789`；`:16-19` MySQL `rosecloud123`；`:40,46` Redis `rosecloud123`；`:78` `NACOS_AUTH_ENABLE=false`；`:79` `NACOS_AUTH_TOKEN` 为固定 base64 默认值。
- 风险：一旦误用于生产，JWT 可被伪造、DB/Redis 弱口令、Nacos 无鉴权可被任意改写配置/服务发现。
- 修复：密钥缺失即 fail-fast；提供强随机默认并文档化「必须覆盖」；Nacos 鉴权默认开启。

### 🟡 Medium

**M1 — Feign 不转发 `Authorization`，与 Javadoc 自相矛盾**
- 证据：`SecurityHeaderFeignPropagator.java:14-26` 只转发 `TENANT_ID`；注释声称「调用方身份在接收端由 bearer token 重建」，但实际未转发 bearer，下游永远是 guest。
- 影响：服务间调用无法传递调用方身份，下游无法做基于调用方的授权；叠加 H2（内部端点无认证）形成「无服务间鉴权」的整体现状。
- 修复：明确设计——要么转发 bearer（保证内部端点仍做校验），要么对内部端点加独立服务间 token，并修正注释。

**M2 — 登录无防爆破**
- 证据：`AuthController` 登录接口在白名单内，`AuthServiceImpl.login:55-75` 无限流、无账号锁定。
- 修复：加失败锁定 / 限流（如 bucket4j）或验证码。

**M3 — docker-compose 将中间件端口暴露宿主机**
- 证据：`:25-26`(3306)、`:41-42`(6379)、`:59-61`(5672/15672)、`:82-85`(8848/9848/38082) 绑定 `0.0.0.0`。
- 修复：仅绑定 `127.0.0.1` 或内网网段。

**M4 — `/actuator/**` 免认证白名单**
- 证据：`GatewaySecurityProperties.java:18`、`MonolithSecurityProperties.java:18`、网关 `application.yml:27` 均把 `/actuator/**` 列入白名单。
- 风险：若暴露 `env`/`heapdump` 等端点存在信息泄漏。
- 修复：生产环境收紧 actuator 暴露项并加鉴权，至少排除 `env`/`heapdump`/`shutdown`。

### 🟢 Low

**L1** `clientIp` 直接信任 `X-Forwarded-For` 首值，可被伪造（`AuthController.java:32-37`，仅用于审计，影响较低）。

**L2** 手动拼接 JSON 错误体，message 未转义，存在注入/破坏响应体风险：`JwtAuthenticationGlobalFilter.java:96-100`、`MonolithJwtFilter.java:93-95`。建议复用 `ApiResponse`/ObjectMapper。

**L3** `NoticeDispatchService` 用 `CompletableFuture.runAsync` 默认 ForkJoinPool，无有界线程池、无失败上报。

**L4** `GlobalExceptionHandler.java:22-26` 将所有 `BizException` 映射为 HTTP 400；登录失败 `BAD_CREDENTIALS` 应为 401。建议按 `ErrorCode` 携带的 HTTP 状态映射。

**L5** `UserController.me()` 未登录时返回 `ApiResponse.ok(null)` 而非 401。

### 关键设计决策（记录原因，便于后续维护）

1. **角色码即 authority（粗粒度）**：`sys_menu.perms` 中已有细粒度权限码（`system:user:add` 等），但当前 JWT 未聚合 perms。本轮采用角色码做粗粒度 RBAC，**未**把 perms 塞进 JWT，以避免在身份解析路径增加聚合查询。细粒度 perms 留作后续轮次。
2. **内部端点用共享密钥而非 caller bearer**：`getAuthInfo`（身份解析）自身就在 `/internal/**`，若 Feign 对内部端点也转发 caller bearer 会形成**无限递归**。因此内部端点一律走共享 `X-RoseCloud-Internal-Key`，外部 `/api/**` 才透传 caller bearer。
3. **自定义 `SecurityContextFilter` 必须在 Spring Security 过滤器链之后运行**：`SecurityContextPersistenceFilter` 在链起点会把 `SecurityContextHolder` 重置为空，故该过滤器顺序设为 `Ordered.LOWEST_PRECEDENCE - 10`，确保写入的身份能存活到 controller 调用、`@PreAuthorize` 才能拿到 authority。
4. **网关不剥离 `Authorization`**：`JwtAuthenticationGlobalFilter` 仅校验并转发原始 bearer 下游，因此 `@PreAuthorize` 可端到端生效。

## 八、优化实施记录（第三轮：细粒度 perms 聚合进 JWT）

### 背景
第一轮（C1）采用**角色码**做粗粒度 RBAC；本轮把 `sys_menu.perms` 中的细粒度权限码（如 `system:user:add`）聚合进 JWT，并在 controller 上改用 `hasAuthority('system:xxx')`，实现「按钮级」权限控制。

### 改动链路

1. **数据模型扩展（record 增加 `perms`）**：`CurrentUser`、`UserAuthInfo`、`AuthUser`、`TokenClaims` 均增加 `List<String> perms`（构造时归一化为不可变拷贝）；`TokenClaims.perms` 缺失时回退空列表（向后兼容旧 token）。
2. **鉴权链路聚合**：
   - `JwtTokenCodec.issue()` 签发时写入 `perms` claim；`parse()` 经 `readPerms()` 安全还原（兼容 gson 反序列化的 `List`/`null`）。
   - `UserRepositoryImpl.loadPerms(userId)`（系统模块新增）：`user_role → role_menu → menu.perms` 取并集（去重、过滤空值），在 `findAuthInfo` 中聚合进 `UserAuthInfo`；构造函数扩展注入 `RoleMenuMapper` + `MenuMapper`（仍由 Spring DI 装配）。
   - `FeignUserRepository` 把 `UserAuthInfo.perms` 透传给 `AuthUser`；`AuthServiceImpl.login` / `refresh` 把 `AuthUser.perms` 写入 `CurrentUser` → 进入 JWT。
3. **请求期还原**：`SecurityContextFilter` 解析 token 后，将 `claims.perms()`（来自 JWT，**非** `getAuthInfo`，确保是登录时快照）与 `user.roles()` 一起作为 `SimpleGrantedAuthority` 写入 `SecurityContextHolder`；`CurrentUser` 也携带 perms。
4. **Controller 细粒度化**：`User`/`Role`/`Menu`/`Tenant`/`Dept`/`DictType`/`DictData`/`Session` 的写/列表操作改用精确权限码；`me()`/`me/*` 仍用 `isAuthenticated()`。权限码严格对齐 `V1.1__init_system_data.sql` 中已有的 `sys_menu.perms`：
   - `system:user:{add,edit,del,list}`、`system:role:{add,edit,perm,list}`、`system:menu:{add,edit,del,list}`、`system:tenant:{open,toggle,list}`、`system:dept:{add,edit,del,list}`、`system:dict:{add,edit,del,list}`（DictData 复用）、`system:session:{list,kick}`、`system:notice:list`。
   - 说明：`Tenant.apply/open`→`system:tenant:open`、`disable/enable`→`system:tenant:toggle`。`Notice.publish`、`Config`、`AuditLog`、`LoginLog` 当时种子无对应权限码，暂保留角色校验——**上述四处已在第四轮补齐为按钮级权限码（见第九节）**。

### 关键设计决策

1. **权限码以 JWT 内嵌快照为准**：请求期不重新聚合 perms（避免每次查菜单树），与 JWT 无状态一致。代价：登录后变更权限需等 token 过期或重新登录/刷新才生效（刷新会重新聚合，因 `refresh` 重新走 `findByUsername`）。
2. **角色与权限码并存为 authority**：`roles` 仍由 `getAuthInfo` 解析（近实时），`perms` 来自 JWT（登录快照），合并进 `SecurityContextHolder`；`@PreAuthorize` 可混用 `hasAuthority('platform-admin')` 与 `hasAuthority('system:user:add')`。
3. **种子权限码是唯一事实源**：所有 `@PreAuthorize` 引用的权限码均能在 `V1.1__init_system_data.sql` 中找到，规避「配了注解却永远 403」。

### 验证
