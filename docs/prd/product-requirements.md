# RoseCloud 需求文档

## 1. 文档目标与范围

### 1.1 文档目标

本文件用于明确 RoseCloud 当前版本的产品范围、业务边界、角色分工与模块级需求，作为后续产品设计、验收设计和文档协同的统一依据。

### 1.2 当前版本范围

| 范围项 | 当前口径 |
| --- | --- |
| 产品定位 | 企业后台与 SaaS 平台底座，带一个轻量示例业务域 |
| 当前版本目标 | 跑通“租户开通 -> 管理员激活 -> 租户运营 -> 用户使用 -> 通知触达 -> 审计留痕”闭环 |
| 当前确认交付重点 | 后端业务闭环、平台管理能力、租户管理能力、通知消费闭环 |
| 不纳入当前版本完整交付 | 公共入口与工作区前端页面、独立任务中心、客户与临时登录令牌、完整外部身份接入、多因素验证 |

### 1.3 范围边界

- 本文档只描述业务需求、角色诉求和验收闭环。
- 本文档不描述实现方式、技术方案、接口设计、数据结构和部署方式。
- 若某项能力在现有仓库中没有明确业务归属或只有参考性线索，则放入“待确认项”，不视为既定需求。

## 2. 产品定位

RoseCloud 面向需要同时支持平台治理和租户运营的企业后台场景，定位为可承载真实业务闭环的平台底座，而不是仅用于演示的空壳框架。

当前版本的产品定位包括：

- 支持平台管理员从平台视角管理租户和平台能力。
- 支持租户管理员在本租户内完成人员、权限、基础配置和通知运营。
- 支持普通业务用户进入被授权的工作范围，接收和处理与自己相关的信息。
- 通过一个轻量示例业务域，验证平台能力能够支撑真实用户闭环。

## 3. 角色定义

| 角色 | 主要职责 | 关注重点 |
| --- | --- | --- |
| 平台管理员 | 管理租户、平台级配置、平台级通知、平台级审计与治理 | 平台整体可控、租户可运营、关键动作可追溯 |
| 租户管理员 | 管理本租户用户、角色、菜单、基础设置和租户内通知 | 本租户可运营、权限边界清晰、信息可触达 |
| 普通业务用户 | 使用被授权功能、查看个人相关信息、完成已读或确认动作 | 登录可用、入口明确、与本人相关的信息处理闭环 |
| 安全管理员 | 关注登录风险、审计记录和关键安全策略 | 登录过程可追踪、关键操作可审计 |

说明：

- 安全管理员在当前版本可由平台管理员兼任。
- 运维、开发、实施等角色属于交付支持角色，不作为当前版本的内置业务角色要求。

角色与租户关系约定：

- 每个用户只有一个所属租户，即其被创建时所在的租户。
- 用户的所属租户是固定的，不因登录、切换租户或代入租户而变化。
- 用户可被授权进入多个成员租户，但成员租户集合不改变其所属租户。
- 当前活动租户是登录态的一部分，只表示本次访问正在使用哪个租户上下文。
- 角色以租户为边界生效，普通角色与内置角色都归属某个租户。

## 4. 总体业务闭环

当前版本围绕以下业务闭环组织：

1. 平台管理员创建租户并为租户指定能力边界。
2. 系统创建租户与首个租户管理员，并将租户置为待激活状态。
3. 租户管理员完成首次激活、设置登录密码，并以租户管理员身份初始化本租户。
4. 租户激活完成后进入启用状态，租户管理员开始维护本租户用户、角色、菜单、部门、字典和基础配置。
5. 普通业务用户登录系统，进入被授权的工作范围并查看个人相关信息。
6. 平台管理员或租户管理员发布公告或通知，面向目标范围触达用户。
7. 普通业务用户完成阅读、已读或确认，形成信息消费闭环。
8. 平台侧持续保留登录记录和关键操作记录，用于审计和运营追踪。

## 5. 模块级需求
## 5.1 认证与授权

### 业务目标

保障平台管理员、租户管理员和普通业务用户能够在各自权限边界内安全进入系统，并维持可管理的登录状态、访问资格和会话控制能力。

### 核心需求功能

- 支持统一的账号密码登录。
- 支持首个管理员激活并完成首次设密。
- 支持用户登录后获取本人可进入的租户列表，列表来源于其所属租户和已授权成员租户。
- 支持记录用户上一次成功进入的活动租户。
- 支持在存在多个可用租户时，由用户选择当前进入的活动租户。
- 支持在存在多个可用租户且用户上一次活动租户仍然可用时，优先直接进入该租户。
- 支持在仅存在一个可用租户时，直接以该租户进入系统。
- 支持登录态绑定当前活动租户，并在后续访问中持续生效。
- 支持用户在已登录状态下切换活动租户。
- 支持用户查看当前个人信息。
- 支持用户修改本人登录密码。
- 支持按角色与菜单控制访问范围。
- 支持在线会话查询和强制下线。
- 支持退出登录并终止当前登录状态。
- 支持租户待激活、启用、停用、禁用状态对登录、进入、切换和读写范围产生不同限制。
- 支持平台管理员与普通租户用户使用不同的进入语义。

### 登录流程

1. 用户输入账号和密码发起登录。
2. 系统校验用户身份，并确定该用户的所属租户。
3. 系统计算该用户当前可进入的成员租户集合。
4. 若用户只有一个可用租户，则系统直接以该租户建立活动登录状态。
5. 若用户有多个可用租户，则系统优先复用用户上一次仍然可用的活动租户。
6. 若没有可复用的活动租户，则系统返回可进入租户列表，由用户选择当前活动租户。
7. 登录成功后，系统把当前活动租户写入登录态，并在后续访问中持续生效。
8. 平台管理员登录后默认进入平台视角，即系统租户 `ROOT`。

### 切换租户流程

1. 用户在已登录状态下发起切换租户操作。
2. 系统先判断该用户是否有权进入目标租户。
3. 普通用户只能切换到其成员租户范围内的租户。
4. 平台管理员可以切换到授权范围内的普通租户，用于代入查询或处理事务。
5. 切换成功后，系统生成新的活动登录状态，新的活动租户成为后续请求的上下文。
6. 原活动登录状态失效，不再继续作为后续访问依据。
7. 切换过程应可追踪，并保留审计记录。

### 多因素验证（MFA）

多因素验证（MFA）是在账号密码第一因素通过后追加的第二因素校验，作为认证与授权模块的后续阶段增强能力。功能设计参考 ThingsBoard 的双因素认证（详见 `docs/adr/thingsboard-reference.md` A3/A5.2），首版以 TOTP 为主要第二因素，受租户套餐可选能力门控。

第二因素以可插拔提供者组织（对齐 ThingsBoard 的 TOTP / SMS / EMAIL / BACKUP_CODE 提供者模型），便于后续扩展新的第二因素类型：

- TOTP：基于时间的一次性密码，兼容标准 TOTP 验证器应用，绑定生成密钥与二维码。
- 恢复码：一次性恢复码，用于丢失 TOTP 设备时恢复访问，使用后即失效。
- 短信/邮件 OTP：一次性密码经短信或邮件投递，属后续增强；投递复用通知中心的 `/notices/send` 事务性发送接口（调用方指定渠道、同步返回结果、敏感内容脱敏存储），不进入站内信收件箱。

核心功能：

- 支持用户为本人账号开启/关闭 MFA，并可配置一个或多个第二因素提供者。
- 支持绑定 TOTP 时生成密钥与二维码，用户扫码录入并输入一次有效验证码完成绑定（生成 -> 提交 -> 校验通过后保存配置）。
- 支持生成一次性恢复码，用于丢失 TOTP 设备时恢复访问；恢复码使用后即失效。
- 支持已启用 MFA 的用户登录时先取得待验证凭证，完成第二因素后再换发正式访问凭证。
- 支持对验证码发送与校验分别限频，并在连续校验失败达到阈值后锁定该用户的 MFA 验证。
- 支持 MFA 开启受租户套餐可选能力门控，未授权该能力的租户下用户不得启用 MFA。
- 支持平台/租户级强制 MFA（后续增强）：被强制用户下次登录须先完成 MFA 配置才能进入正式访问。

### 多因素验证流程

1. 用户输入账号和密码发起登录。
2. 系统校验账号密码通过。
3. 若该用户已启用 MFA，系统不发放正式访问凭证，而是发放待验证凭证，并提示需要第二因素。
4. 用户提交 TOTP 验证码或恢复码。
5. 系统校验第二因素：通过则换发正式访问凭证（绑定活动租户，与普通登录一致）；不通过则保持待验证状态，按限频策略允许重试。
6. 待验证凭证短期有效且仅限完成第二因素验证动作，不得用于正式业务访问。

### MFA 绑定流程

1. 已登录用户在个人安全设置发起绑定 MFA。
2. 系统校验该用户所属租户已开放 MFA 可选能力。
3. 系统生成 TOTP 密钥与二维码并展示给用户。
4. 用户使用 TOTP 验证器应用扫码，并输入一次有效验证码。
5. 系统校验通过后完成绑定，生成并一次性展示恢复码供用户妥善保存；此后该用户登录需完成第二因素。

### 涉及角色

| 角色 | 关注点 |
| --- | --- |
| 平台管理员 | 平台登录、平台视角访问、在线会话治理 |
| 租户管理员 | 首次激活、日常登录、租户范围访问 |
| 普通业务用户 | 快速登录、访问本人被授权功能 |
| 安全管理员 | 登录风险可追踪、会话可管理 |

### 登录态与凭证类型

当前版本及后续扩展中，认证相关凭证需按业务语义分层管理：

| 凭证类型 | 用途定位 | 当前口径 |
| --- | --- | --- |
| 正式访问凭证 | 用户已完成完整进入流程后访问正式业务范围 | 应作为正式登录态使用，并绑定当前活动租户 |
| 续期凭证 | 用于续期正式登录态 | 应仅用于续期，不直接承载正式业务访问 |
| 激活凭证 | 用于首个管理员激活与首次设密 | 只服务激活流程，不进入正式业务访问范围 |
| 待选租户凭证 | 用于身份校验完成但尚未确定活动租户的过渡阶段 | 只允许完成租户选择相关动作，不进入正式业务访问范围 |
| 待验证凭证 | 用于未来多因素验证等增强认证流程中的中间状态 | 只允许完成对应验证动作，不进入正式业务访问范围 |
| 代入访问凭证 | 用于平台管理员进入目标租户范围处理事务 | 应与普通正式访问状态区分，并保持可审计 |
| 客户临时访问凭证 | 用于未来客户临时访问场景 | 应与普通正式访问状态区分，并保持短期、受限、可失效 |

补充规则：

- 正式访问类凭证应承载正式访问所需的身份与活动租户语义。
- 所属租户用于标识用户最初被创建在哪个租户。
- 成员租户用于标识用户被授权可以进入哪些租户。
- 活动租户用于标识当前登录态正在使用哪个租户上下文。
- 流程型临时凭证可采用短期、一次性或受限形式存在，只服务于特定流程动作。
- 任何流程型临时凭证都不得直接放行正式业务访问范围。
- 未来引入外部身份接入、多因素验证、代入租户和客户临时访问时，应沿用“正式访问态”和“流程中间态”分层规则，不得混用同一类凭证语义。

### 典型用户用例

| 触发场景 | 用户动作 | 系统结果 |
| --- | --- | --- |
| 新租户处于待激活状态，首个管理员需要完成激活 | 租户管理员通过激活入口设置密码并提交 | 系统完成激活并允许其进入租户初始化流程 |
| 已有用户需要进入系统，但没有任何可用租户 | 用户输入账号和密码发起登录 | 系统不建立正式业务访问状态，并明确提示当前无法进入系统 |
| 已有用户需要进入系统，且只存在一个可用租户 | 用户输入账号和密码发起登录 | 系统校验身份后，直接以该租户建立登录状态 |
| 已有用户需要进入系统，且上一次活动租户仍然可用 | 用户输入账号和密码发起登录 | 系统校验身份后，直接以上一次活动租户建立登录状态 |
| 已有用户需要进入系统，且存在多个可用租户 | 用户输入账号和密码发起登录 | 系统校验身份后返回可进入租户范围，等待用户选择活动租户 |
| 用户需要切换当前工作的租户 | 用户在已登录状态下选择另一个可用租户 | 系统校验切换资格后，以新租户建立新的活动登录状态，并使原活动登录状态失效 |
| 用户需要确认自己的登录身份和归属 | 用户进入个人信息入口 | 系统展示当前用户的基础信息和当前归属范围 |
| 用户需要更新个人密码 | 用户提交旧密码和新密码 | 系统更新密码并使后续访问基于新密码生效 |
| 平台侧发现异常会话 | 平台管理员查看在线会话并执行强制下线 | 系统终止指定会话，该会话后续不再有效 |
| 用户尝试进入一个已禁用租户 | 用户登录后选择该租户，或在已登录状态下切换到该租户 | 系统拒绝进入或切换，不允许以该租户建立活动登录状态 |
| 用户当前活动租户已停用 | 用户继续访问系统 | 系统允许其继续以该租户查看允许范围内的信息，但限制新增、修改、删除等写操作 |
| 平台管理员需要处理平台事务 | 平台管理员完成登录 | 系统默认以平台视角进入，而不是直接进入某个普通租户 |
| 用户希望为本人账号增加第二因素保护 | 已登录用户在安全设置发起绑定 MFA | 系统校验租户能力后生成 TOTP 二维码，用户录入有效验证码后完成绑定并下发恢复码 |
| 已启用 MFA 的用户登录 | 用户输入账号密码后再输入 TOTP 验证码 | 系统先发放待验证凭证，校验第二因素通过后换发正式访问凭证并绑定活动租户 |
| 用户丢失 TOTP 设备 | 用户在登录第二因素环节输入恢复码 | 系统校验恢复码通过后换发正式访问凭证，该恢复码即失效 |
| 用户希望关闭 MFA | 已登录用户在安全设置关闭 MFA | 系统解除绑定并停用第二因素，该用户后续登录不再要求第二因素 |
| 未开放 MFA 能力的租户下用户尝试绑定 | 用户在安全设置发起绑定 MFA | 系统拒绝绑定，提示当前租户未开放该能力 |

### 约束与边界

- 主归属模块为“认证与授权”，次级关联模块为“系统管理”“安全与审计”。
- 当前版本确认的正式登录主路径为账号密码登录。
- 当用户存在多个可用租户且尚未确定本次活动租户时，系统不得放行其进入正式业务访问范围。
- 活动租户属于正式登录状态的一部分，不能只作为单次访问中的临时信息存在。
- 用户上一次活动租户只能在其本次仍然可用时被直接复用。
- 用户可进入的租户范围必须先经过系统校验，再允许其选择和切换。
- 用户切换活动租户成功后，原活动登录状态失效，后续访问以新的活动租户为准。
- 用户所属租户不随登录态切换而变化。
- 本模块中的“可用租户”指当前允许用户进入并建立活动登录状态的租户范围；当前版本下，启用租户属于可用租户，停用、禁用、待激活租户不属于可用租户。
- 待激活租户不得作为正式登录目标、进入目标或切换目标。
- 已禁用租户不得作为登录目标、进入目标或切换目标。
- 已停用租户可保留受限访问范围，但不得提供新增、修改、删除等写操作。
- 平台管理员默认进入平台视角；如需处理普通租户事务，再进入目标租户范围。
- 外部身份接入、不依赖密码的其他登录方式不属于当前版本既定范围；多因素验证为后续阶段既定增强（见「多因素验证（MFA）」），不纳入当前版本主路径。
- MFA 在账号密码第一因素通过后追加第二因素，不替代第一因素。
- 待验证凭证遵循流程型临时凭证规则，短期、仅限完成第二因素验证动作，不得放行正式业务访问。
- 待验证凭证应短期有效（参考 ThingsBoard 默认 30 分钟）且为单作用令牌，不含续期凭证，校验通过或超时后即失效。
- TOTP 为首版 MFA 既定第二因素；短信/邮件 OTP 属后续增强（投递复用通知中心 `/notices/send`，不进入首版 MFA 实现），不纳入首版 MFA。
- MFA 的开启受租户套餐可选能力门控，未授权该能力的租户下用户不得启用 MFA。
- MFA 的校验、待验证凭证与恢复码管理收口在认证与授权模块，与正式访问态、会话治理边界一致。
- TOTP 密钥与恢复码应安全存储：密钥加密落库，恢复码仅存哈希，明文不在系统持久化。
- 本模块负责“谁可以进入系统、进入后可保持何种访问资格”，不负责菜单定义、角色维护和组织管理本身。

### 分阶段说明

| 阶段 | 范围 |
| --- | --- |
| 当前版本 | 登录、激活、登录后选租户、记忆并复用上次活动租户、切换活动租户、按租户状态限制访问范围、改密、当前用户信息、在线会话管理、强制下线 |
| 后续阶段 | 多因素验证（见「多因素验证（MFA）」，首版 TOTP + 恢复码 + 待验证凭证登录流程 + 发送/校验限频与失败锁定，受套餐能力门控；强制 MFA、短信/邮件 OTP、记住设备为再后续增强）、外部身份接入、更细粒度的会话治理（范围待确认） |

## 5.2 安全与审计

### 业务目标

确保登录行为、关键操作和关键状态变化具备可追踪性，满足平台治理、风险排查和责任留痕的基本要求。

### 核心需求功能

- 支持登录记录查询。
- 支持关键操作记录查询。
- 支持关键动作的操作人、时间和结果留痕。
- 支持围绕租户、用户和动作维度的追踪。
- 支持安全相关关键动作的审计留痕。

### 涉及角色

| 角色 | 关注点 |
| --- | --- |
| 平台管理员 | 平台关键操作可追溯 |
| 安全管理员 | 登录过程和风险动作可排查 |
| 租户管理员 | 与本租户有关的关键操作可被追踪 |

### 典型用户用例

| 触发场景 | 用户动作 | 系统结果 |
| --- | --- | --- |
| 平台需要排查一次异常登录 | 管理员查询登录记录 | 系统返回登录结果和关联上下文，支持进一步排查 |
| 平台需要核对谁执行了某项关键操作 | 管理员查询操作记录 | 系统返回操作人、动作、时间和结果 |
| 平台需要追踪租户开通过程 | 管理员按租户筛选关键记录 | 系统返回与该租户相关的关键操作链路 |
| 平台需要核对通知或权限调整是否被执行 | 管理员查询操作审计 | 系统返回对应动作的留痕信息 |

### 约束与边界

- 主归属模块为“安全与审计”，次级关联模块为“认证与授权”“租户管理”“系统管理”“通知中心”。
- 当前版本确认的是登录记录和关键操作记录的基础审计能力。
- 本模块不等同于独立安全运营中心，也不包含复杂告警联动、风险评分和自动处置。
- 更复杂的安全策略治理和安全事件闭环，当前待确认。

### 分阶段说明

| 阶段 | 范围 |
| --- | --- |
| 当前版本 | 登录记录、关键操作记录、基础追踪能力 |
| 后续阶段 | 更复杂的安全事件治理与自动处置，范围待确认 |

## 5.3 配置管理

### 业务目标

让平台管理员能够通过统一的配置项目录维护平台级配置边界与可选能力开关，让用户能够管理个人偏好，并让平台以可控方式决定哪些能力当前可用、哪些能力当前关闭。配置模型借鉴 ThingsBoard 的“先分配置域、再定数据形态、再定继承规则”思路（见 `docs/adr/thingsboard-reference.md` Part C），但在当前仓库中落地为三层通用键值承载：`sys_setting_key`（配置项定义）、`sys_system_setting`（平台级配置）、`sys_user_setting`（个人设置）。

### 配置管理模型（当前仓库可落地）

| 配置层 | 数据载体 | 表 / 实体 | 当前字段 | 业务归属与生效范围 | 可管理的配置类别 | 与 ThingsBoard 对照 |
| --- | --- | --- | --- | --- | --- | --- |
| 配置项定义 | `SettingKey` | `sys_setting_key` | `key`、`name`、`remark` | 定义“有哪些配置可被管理”，不存值、不区分归属层级；全平台可见的配置边界清单 | 配置项目录：注册一项可被维护的配置（如 `system.mail.host`、`feature.notice.sms.enabled`），作为统一配置边界的可见清单 | ThingsBoard 无独立配置项目录，RoseCloud 以 `SettingKey` 显式登记配置项，便于前端展示与权限收口 |
| 平台级配置 | `SystemSetting` | `sys_system_setting` | `settingKey`（主键）、`value` | 平台维度、全局唯一键；值对**所有租户、所有用户**共享，仅平台管理员维护 | 平台级参数（字符串值或 JSON 字符串）；可选能力的启用/关闭（布尔型 `feature.*.enabled`） | 对应 ThingsBoard `AdminSettings`（系统租户级 `key + jsonValue`），本仓库当前值为 `String`，复杂结构以 JSON 字符串承载 |
| 个人设置 | `UserSetting` | `sys_user_setting` | `userId`、`settingKey`、`value` | 用户维度、按 `userId` 隔离；值仅对**该用户本人**生效，不影响他人 | 个人偏好（按用户维度的 `key → value`，如主题、语言、默认视图、通知偏好） | 对应 ThingsBoard `UserSettings`（`userId + type + JsonNode`），本仓库以 `userId + settingKey` 承载，等价“按 key 分组的个人偏好” |
| 默认与回退 | `SettingKey` + `SystemSetting` | 跨表 | — | 定义层登记默认值，读取时回退；回退结果按平台级生效 | 配置项可携带默认值；未显式设置平台级配置时回退到默认值 | ThingsBoard 各配置域自带默认值与回退（如 `UserNotificationSettings` 默认允许），本仓库以“定义默认值 + 读取时回退”对齐 |

> 同一类偏好可同时存在两级：例如“通知渠道”在 `sys_system_setting` 决定平台是否开放短信能力（平台级），在 `sys_user_setting` 决定某用户是否想收短信（个人级）。值的业务类型（布尔开关 / 数字 / JSON）不是表结构约束，而是读写方约定——列均为 `String`。

> 说明：当前 `SystemSetting.value` 为 `String`，可存放布尔、数字、JSON 等。若某配置域字段语义稳定且结构较大（如租户画像 `TenantProfile`、套餐 `TenantPackage`），应按 ThingsBoard “强类型配置对象”思路单独建模，不塞进通用键值表——租户画像与套餐已在 5.4/5.5 独立建模。

### 核心需求功能

- 支持配置项目录管理：定义、维护、查看配置项（`SettingKey` 的增改查删）。
- 支持平台级配置项维护：按 `settingKey` 读写平台级参数（`SystemSetting`）。
- 支持个人设置管理：用户按 `settingKey` 读写本人偏好（`UserSetting`）。
- 支持按能力边界启用或关闭可选能力：以平台级 `feature.*.enabled` 等配置项承载开关，其他模块读取后限制能力是否可用。
- 支持默认能力边界和默认配置约束：配置项可携带默认值，未显式设置时回退默认。
- 支持配置变更的审计留痕（关键配置写操作进入安全与审计闭环）。

### 涉及角色

| 角色 | 关注点 |
| --- | --- |
| 平台管理员 | 配置项目录可维护、平台级配置可维护、能力边界可控制 |
| 租户管理员 | 与租户运营相关的边界可被理解和使用（当前版本以平台级配置为主，租户级开关为后续能力） |
| 普通业务用户 | 个人偏好可查看、可维护 |

### 典型用户用例

| 触发场景 | 用户动作 | 系统结果 |
| --- | --- | --- |
| 平台需要新增一项可维护配置 | 平台管理员在配置项目录登记新配置项（key/name/remark，可选默认值） | 系统建立 `SettingKey`，该配置项进入统一配置边界清单，允许后续按规则读写 |
| 平台需要调整某项全局参数 | 平台管理员修改平台级配置 `settingKey` 的值 | 系统写入/更新 `SystemSetting`，后续读取方按最新值生效 |
| 平台需要关闭某项可选能力 | 平台管理员将 `feature.<能力>.enabled` 置为 `false` | 系统更新平台级配置；相关模块读取后限制该能力是否可用（如关闭短信通道则通知中心不开放短信投递） |
| 平台需要开启某项可选能力 | 平台管理员将 `feature.<能力>.enabled` 置为 `true` | 系统更新平台级配置；相关模块读取后放开该能力 |
| 用户需要保存个人偏好 | 普通业务用户更新个人设置（如主题、语言、默认视图） | 系统写入该用户的 `UserSetting`，后续访问按本人偏好呈现 |
| 用户需要查看个人偏好 | 普通业务用户查询本人设置列表 | 系统返回该用户的 `UserSetting` 集合 |
| 平台需要查看当前配置边界 | 平台管理员查看配置项目录与平台级配置现状 | 系统展示全部 `SettingKey` 及对应的 `SystemSetting` 值，配置边界可见、可审计 |
| 平台需要确认某项配置是否缺失 | 平台管理员查询某 `settingKey` 是否存在平台级配置 | 系统返回该配置项值；若不存在且定义了默认值则回退默认值，否则提示未配置 |
| 平台管理员修改关键配置 | 平台管理员保存/删除平台级配置或配置项 | 系统记录审计日志（操作人、时间、前后值），可供安全与审计追溯 |
| 租户管理员希望理解本租户能力边界 | 租户管理员查看平台级能力开关与套餐结果 | 系统展示当前生效的能力边界（只读；租户级开关为后续能力，见约束） |
| 某能力变更需联动其他模块 | 平台管理员关闭通知短信能力 | 通知中心读取平台级开关后，不再向用户开放短信投递方式，表现为明确不可用而非运行时不确定 |

### 约束与边界

- 主归属模块为“配置管理”，次级关联模块为全部业务模块。
- 当前版本确认的配置项目录、平台级配置、个人设置和能力边界控制需求已经成立，且已有对应数据表与服务（`SettingKeyService`/`SystemSettingService`/`UserSettingService`）。
- 平台级配置与能力开关采用通用键值承载（`key → String value`）；字段语义稳定、结构较大的配置域（租户画像、套餐）应单独强类型建模，不进入通用键值表。
- 配置项值当前为 `String`，可承载布尔/数字/JSON；复杂 JSON 结构建议以 JSON 字符串存储，不在当前版本引入独立的 JSON 列类型。
- 租户级能力开关（按租户维度覆盖/回退平台配置）当前为后续能力：现有 `SystemSetting` 为平台级、不带租户维度，租户级开关粒度与继承规则待确认。
- 统一能力查询入口（前端一次性拉取当前可用能力集合）当前为后续能力，待确认。
- 本模块不负责租户生命周期管理，也不替代系统管理中的人员和授权治理；租户画像与套餐边界分别在 5.4/5.5 建模。

### 分阶段说明

| 阶段 | 范围 |
| --- | --- |
| 当前版本 | 配置项目录（定义/维护/查看）、平台级配置读写、个人设置读写、以平台级 `feature.*.enabled` 承载的能力边界控制、默认值与回退、关键配置变更审计留痕 |
| 后续阶段 | 统一能力查询入口、租户级能力开关与继承规则、更细粒度能力包、复杂配置域的强类型化，范围待确认 |


## 5.4 租户管理

### 业务目标

支撑平台管理员完成租户创建、状态管理、运营资格控制和跨租户治理，使平台能够同时服务多个租户，并明确区分“待激活”“可使用”“可运营”“只读受限”和“不可用”等业务状态。

### 核心需求功能

- 多租户作为当前版本默认启用的主路径能力。
- 支持创建租户，且租户开通等同于创建成功。
- 支持查看租户列表和租户详情。
- 支持维护租户基础信息。
- 支持租户待激活、启用、停用、禁用四种状态管理。
- 支持租户有效期管理，并在到期后自动进入停用状态。
- 支持租户是否可被进入、可被切换的状态控制。
- 支持用户上一次活动租户在仍然可用时被再次进入。
- 支持在租户创建时生成首个管理员激活流程。
- 支持租户创建阶段仅创建租户和首个租户管理员。
- 支持首个管理员激活成功后，以租户管理员身份初始化该租户的角色、菜单等基础内容。
- 支持首个管理员激活成功后，将租户从待激活状态变更为启用状态，并进入正式可运营状态。
- 支持平台管理员从平台视角管理全部租户。
- 支持平台管理员代入目标租户进行只读查看，用于排查问题。
- 支持同一用户在被授权的多个租户之间切换，并按目标租户权限执行操作。

### 涉及角色

| 角色 | 关注点 |
| --- | --- |
| 平台管理员 | 租户创建、租户状态、租户可运营性、跨租户治理 |
| 租户管理员 | 首次激活与租户初始化、租户可运营状态、被授权租户切换 |
| 安全管理员 | 租户状态变化是否可追踪 |

### 租户状态规则

| 状态 | 业务含义 | 登录与进入 | 切换 | 写操作 |
| --- | --- | --- | --- | --- |
| 待激活 | 租户已创建，但尚未完成首个管理员激活和租户初始化 | 不允许正式登录与进入 | 不允许 | 不允许 |
| 启用 | 租户处于正常可用状态 | 允许 | 允许 | 允许 |
| 停用 | 租户处于受限可用状态，常见触发原因为有效期到期 | 允许 | 允许进入该租户的受限范围 | 不允许 |
| 禁用 | 租户处于不可用状态 | 不允许 | 不允许 | 不允许 |

补充规则：

- 租户创建成功即视为已开通，但初始状态为待激活。
- 待激活阶段仅完成租户和首个租户管理员的建立，不完成租户内基础内容初始化。
- 待激活阶段只允许首个租户管理员完成激活和初始化相关动作，不允许开展正式租户运营。
- 待激活阶段允许平台管理员查看该租户的基础信息和当前状态，但不允许以该租户开展运营操作。
- 首个管理员激活成功后，系统以租户管理员身份完成租户初始化，并将租户置为启用状态。
- 若首个管理员已完成身份激活，但租户初始化尚未完成，租户仍保持待激活状态，不进入启用状态。
- 租户到期后自动进入停用状态，不再保持完整运营能力。
- 停用强调“受限使用”，禁用强调“完全不可用”。

### 典型用户用例

| 触发场景 | 用户动作 | 系统结果 |
| --- | --- | --- |
| 平台引入一个新租户 | 平台管理员录入租户信息并提交创建 | 系统创建租户和首个租户管理员，并将租户置为待激活状态 |
| 平台管理员需要确认待激活租户是否已准备就绪 | 平台管理员查看该租户详情 | 系统展示该租户基础信息、当前状态和首个管理员激活进度，但不允许执行租户运营动作 |
| 首个管理员完成首次激活 | 租户管理员通过激活入口设置密码并提交 | 系统完成该租户的初始化，并将租户变更为启用状态和正式可运营状态 |
| 首个管理员已完成身份激活，但租户初始化未完成 | 租户管理员再次进入激活初始化流程 | 系统保持该租户为待激活状态，直到初始化完成后再进入启用状态 |
| 某租户超过有效期 | 平台管理员或租户管理员继续进入该租户范围 | 系统自动按停用状态处理，允许查看受限范围内的信息，但限制写操作 |
| 平台需要彻底阻断某租户继续使用 | 平台管理员执行禁用 | 系统禁止该租户后续登录、进入和切换，并保留状态变化记录 |
| 已停用租户恢复正常运营 | 平台管理员执行启用 | 系统恢复该租户完整可用状态 |
| 用户再次登录系统时，上一次活动租户仍然可用 | 用户发起登录 | 系统允许该用户直接再次进入该租户范围 |
| 平台管理员需要排查某租户问题 | 平台管理员从平台视角代入目标租户 | 系统允许其只读查看目标租户范围内信息，但不允许写操作 |
| 同一用户同时负责多个租户 | 用户在已登录状态下切换目标租户 | 系统校验其在目标租户的授权关系后，按目标租户权限建立新的活动范围 |

### 约束与边界

- 主归属模块为“租户管理”，次级关联模块为“认证与授权”“系统管理”“租户套餐”“配置管理”“安全与审计”。
- 本模块负责租户这一平台级对象的生命周期与边界管理，不负责租户内具体人员、角色、菜单和字典的日常维护。
- 租户开通等同于租户创建，不单独定义为第二条业务流程。
- 当前版本以多租户为默认启用的主路径能力。
- 租户状态不仅影响租户运营，也影响用户是否能够进入或继续停留在该租户范围内。
- 用户上一次活动租户是否可再次直接进入，仍取决于该租户在本次登录时是否可用。
- 待激活租户仅用于首个管理员完成激活和初始化，不进入正式业务访问范围。
- 待激活阶段允许查看状态，不允许进行用户、角色、菜单、通知和配置等正式运营动作。
- 平台管理员可查看待激活、停用、禁用租户的基础信息和状态，但只读代入仅适用于启用或停用租户；待激活和禁用租户不进入租户视角。
- 停用租户允许受限访问但不允许写操作；禁用租户不允许登录、进入和切换。
- 首个管理员激活成功并完成租户初始化，是租户进入启用状态和正式可运营状态的确认条件。
- 平台管理员可从平台视角查看全部租户，并可代入目标租户执行只读排查；该能力不等同于无边界代操作。
- 只有同时被授权到多个租户的用户，才允许在这些租户之间切换；该规则不意味着任意租户管理员都可跨租户操作。
- 租户套餐负责承载能力画像、有效期和规模边界，本模块只负责引用套餐结果并管理租户状态。
- 商业计费、账务结算和无边界跨租户写操作不作为当前版本既定范围。

### 分阶段说明

| 阶段 | 范围 |
| --- | --- |
| 当前版本 | 租户创建后进入待激活、首个管理员激活并初始化租户后进入启用、有效期到期后自动停用、待激活/启用/停用/禁用状态控制、按状态限制进入与切换、平台管理员只读代入查看 |
| 后续阶段 | 更复杂的续期规则、复杂配额治理和更细粒度的跨租户治理，范围待确认 |

## 5.5 租户套餐

### 业务目标

通过租户套餐统一承载租户的能力画像、有效期和基础规模边界，使平台能够按租户类型控制可用范围，同时避免将套餐扩展为复杂商业计费系统。

### 核心需求功能

- 支持平台管理员定义租户套餐。
- 支持套餐作为租户能力画像的正式载体。
- 支持为租户绑定当前生效套餐。
- 支持套餐定义租户有效期。
- 支持套餐定义租户可使用的能力范围。
- 支持套餐定义租户基础运营规模边界。
- 支持套餐变化影响租户当前可用边界。
- 支持套餐到期与租户状态联动。

### 涉及角色

| 角色 | 关注点 |
| --- | --- |
| 平台管理员 | 套餐可定义、可绑定、可调整，并能约束租户边界 |
| 租户管理员 | 本租户当前套餐边界清晰、可理解 |
| 安全管理员 | 套餐变化对租户状态和能力边界的影响可追踪 |

### 套餐能力边界明细

| 边界对象 | 当前版本要求 |
| --- | --- |
| 租户有效期 | 套餐应明确租户的有效使用期限，并与租户状态联动 |
| 模块可用范围 | 套餐应明确租户可使用哪些租户侧模块，至少覆盖系统管理、通知中心等面向租户运营和使用的能力范围 |
| 用户规模边界 | 套餐应明确租户可承载的用户规模上限，并对新增用户产生约束 |
| 组织规模边界 | 套餐应明确租户可维护的部门等组织对象规模边界，并对新增组织对象产生约束 |
| 通知使用边界 | 套餐应明确租户是否可使用通知发布与通知消费相关能力，以及对应的使用范围 |
| 配置与可选能力边界 | 套餐应明确租户是否可使用特定可选能力和配置范围，不在套餐范围内的能力不应对该租户开放 |

补充规则：

- 套餐边界面向租户侧运营与使用范围，不用于定义平台管理员的基础管理权限。
- 套餐边界应能被租户管理员理解和查看，而不是只作为平台内部概念存在。
- 套餐调整后，租户当前可用边界应按最新套餐结果变化。
- 套餐到期、套餐调整和套餐切换都应影响租户当前状态或可用范围。
- 用户规模和组织规模超出套餐边界时，应禁止继续新增，但不影响既有存量对象继续保留。
- 模块、通知和配置能力不在套餐边界内时，租户不应获得对应使用资格。
- 套餐边界导致的能力限制应在租户侧表现为明确不可用，而不是在用户操作后才形成不确定结果。

### 典型用户用例

| 触发场景 | 用户动作 | 系统结果 |
| --- | --- | --- |
| 平台需要为不同类型租户设定不同边界 | 平台管理员创建多个租户套餐 | 系统形成可供租户绑定的套餐集合 |
| 新租户创建时需要明确能力范围 | 平台管理员为该租户绑定套餐 | 系统按套餐定义确定该租户的有效期、能力范围和基础规模边界 |
| 某租户的服务范围需要调整 | 平台管理员调整该租户当前套餐 | 系统按最新套餐更新该租户的能力边界 |
| 某租户套餐到期 | 平台管理员查看租户状态或租户继续使用系统 | 系统按套餐到期规则联动该租户进入停用状态 |
| 租户管理员需要确认当前边界 | 租户管理员查看本租户套餐信息 | 系统展示当前套餐对应的有效期和能力边界 |
| 租户达到套餐允许的用户规模边界 | 租户管理员继续新增用户 | 系统拒绝超出套餐边界的新增动作，并保留现有存量范围 |
| 租户达到套餐允许的组织规模边界 | 租户管理员继续新增部门等组织对象 | 系统拒绝超出套餐边界的新增动作，并保留现有存量范围 |
| 某租户不在通知能力开放范围内 | 租户管理员进入通知相关范围 | 系统按当前套餐边界限制其使用对应能力 |
| 某租户不在特定模块或配置能力开放范围内 | 租户管理员进入对应模块或配置范围 | 系统按当前套餐边界限制其使用对应能力 |

### 约束与边界

- 主归属模块为“租户套餐”，次级关联模块为“租户管理”“配置管理”“安全与审计”。
- 本模块负责套餐这一平台级边界对象，不负责订单、支付、账单、发票和自动扣费。
- 当前版本确认套餐至少承载能力画像、有效期和基础规模边界。
- 套餐变化应影响租户当前的能力边界和状态判断，但不等同于复杂商业化续费体系。
- 套餐如何细分为更复杂的能力包、阶梯配额和精细商业规则，当前不作为既定范围。

### 分阶段说明

| 阶段 | 范围 |
| --- | --- |
| 当前版本 | 套餐定义、套餐绑定、有效期、能力画像、基础规模边界、与租户状态联动 |
| 后续阶段 | 更复杂的套餐组合、续期规则、配额治理和商业化规则，范围待确认 |

## 5.6 系统管理

### 业务目标

为平台管理员和租户管理员提供开展日常运营所需的基础管理能力，覆盖人员、角色、菜单、部门、字典等核心管理对象。

### 核心需求功能

- 支持用户管理。
- 支持角色管理。
- 支持菜单管理。
- 支持用户与角色关系维护。
- 支持角色与菜单关系维护。
- 支持部门管理。
- 支持字典类型与字典数据管理。
- 支持用户获取本人可见菜单。

### 涉及角色

| 角色 | 关注点 |
| --- | --- |
| 平台管理员 | 平台级基础管理能力完整可用 |
| 租户管理员 | 本租户人员和授权关系可维护 |
| 普通业务用户 | 能看到本人被授权的功能入口 |

### 典型用户用例

| 触发场景 | 用户动作 | 系统结果 |
| --- | --- | --- |
| 租户需要新增员工 | 管理员创建用户并分配角色 | 系统生成用户并建立其访问资格 |
| 某个岗位职责发生变化 | 管理员调整角色拥有的菜单范围 | 系统更新该角色的可访问范围 |
| 某位员工岗位变化 | 管理员重新分配用户角色 | 系统刷新该用户的功能访问范围 |
| 租户组织结构调整 | 管理员维护部门层级 | 系统保留最新部门结构，供后续用户归属和管理使用 |
| 用户登录后进入工作范围 | 用户查看本人可见入口 | 系统只展示其被授权的菜单范围 |

### 约束与边界

- 主归属模块为“系统管理”，次级关联模块为“认证与授权”“安全与审计”。
- 本模块负责平台日常管理对象，不负责租户生命周期、登录过程和通知触达本身。
- 当前版本确认的是基础用户、角色、菜单、部门、字典闭环。
- 更细粒度的权限点治理、数据范围治理和更复杂的组织治理不作为当前版本既定范围。

### 分阶段说明

| 阶段 | 范围 |
| --- | --- |
| 当前版本 | 用户、角色、菜单、部门、字典及其基础授权关系 |
| 后续阶段 | 更细粒度权限点、数据范围、复杂组织治理，范围待确认 |

## 5.7 通知中心

### 业务目标

为平台管理员和租户管理员提供统一的信息发布与触达能力，支持模板化发布、规则自动化触发和用户接收偏好，并让普通业务用户能够完成查看、已读和确认，形成通知发布、投递、消费与结果追踪的完整闭环。

### 核心需求功能

**基础闭环**

- 支持平台级和租户级通知发布，支持立即发布与定时发布。
- 支持按目标范围投放（USER / ROLE / TENANT / GLOBAL），目标范围由调用方解析为收件人后传入通知服务。
- 支持通知列表查询与“我的通知”查看，含未读数。
- 支持通知已读、全部已读、确认（需确认通知）与删除。
- 支持投递结果可追踪（受理成功 / 失败 / 超时 / 跳过），并支持按通知请求维度聚合投递统计（成功 / 失败 / 跳过 / 超时计数，按渠道分组）。
- 支持发布前预览：渲染各渠道内容并预估收件人数，发布前校验文案与渠道适配。

**事务性发送**

- 支持事务性发送（如多因素验证的一次性密码、验证码）：由调用方指定单一投递渠道（邮件/短信等）并同步获取投递结果。
- 事务性发送不生成站内信（不进入“我的通知”），且敏感内容（如验证码）脱敏存储，不落明文。
- 多因素验证的短信/邮件 OTP 投递复用此能力（详见「认证与授权 / 多因素验证」）。

**通知模板中心**

- 支持通知模板管理（平台级与租户级）。
- 支持按渠道配置模板内容（标题与正文，含变量占位）。
- 支持发布时选择模板并传入变量进行渲染，或直写内容。

**通知规则自动化**

- 支持通知规则管理（平台级与租户级）。
- 支持事件驱动的规则触发（租户开通、用户激活、改密等）。
- 规则触发时由触发方传入已解析的收件人，通知服务不反查用户数据。

**用户通知偏好**

- 支持用户按通知类型 × 渠道设置接收偏好。
- 投递前按偏好过滤渠道，用户关闭的渠道不实际投递。

**通知渠道与配置**

- 首版支持站内（拉取式，作为默认常开渠道，无需额外配置）与邮件（SMTP）；短信及 Push / Webhook / 钉钉 / 企业微信机器人等作为后续渠道扩展。
- 站内信是通知的一种投递渠道（STATION 渠道）：用户在“我的通知”中查看的是站内信；邮件/短信/Push/Webhook 等渠道的投递结果作为回执记录，不进入“我的通知”。
- 渠道是否可用由渠道配置决定：平台提供默认渠道配置，租户可覆盖；投递时按 `租户配置 -> 平台默认` 解析。平台渠道（如站内信）默认常开，邮件/短信等需管理员配置后才启用。
- 短信与邮件支持多家提供商（如阿里云 / 腾讯云 / SMTP / 第三方 API），按租户配置选择具体提供商。
- 渠道提供商凭证加密存储，不明文落库；投递内容中涉及敏感信息（如验证码）时脱敏存储，查询与展示同样脱敏。
- 支持查询当前租户可用的渠道列表，供发布表单与通知中心限定可选项。
- 支持渠道配置测试：保存配置后可发送测试通知验证可用性。

### 涉及角色

| 角色 | 关注点 |
| --- | --- |
| 平台管理员 | 平台范围的信息发布、模板与规则治理、触达效果 |
| 租户管理员 | 租户内部的信息发布、模板与规则配置、可见范围 |
| 普通业务用户 | 与本人相关的信息及时可见、可处理，并可管理自身接收偏好 |

### 典型用户用例

**平台管理员**

| 触发场景 | 用户动作 | 系统结果 |
| --- | --- | --- |
| 平台需要向全平台或指定范围发布通知 | 创建并立即发布通知 | 系统将通知投放到目标范围，相关用户可见 |
| 平台需要发布跨租户的系统通知 | 发布系统通知 | 系统使用平台级渠道配置投递，相关用户可见 |
| 平台需要规范通知文案 | 创建/维护平台级模板 | 模板可供平台与租户发布时引用 |

**租户管理员**

| 触发场景 | 用户动作 | 系统结果 |
| --- | --- | --- |
| 租户需要发布内部公告 | 创建租户内通知 | 系统只向该租户的目标范围展示相关内容 |
| 租户需要自定义通知文案 | 创建租户级模板 | 模板仅在该租户范围内可用 |
| 租户需要使用自己的短信或邮件提供商 | 配置本租户渠道提供商 | 系统保存租户级渠道配置，投递时按该配置选择提供商，缺失时回退平台默认 |

**平台管理员 / 租户管理员**

| 触发场景 | 用户动作 | 系统结果 |
| --- | --- | --- |
| 需要在指定时间发布通知 | 创建并设定定时发布 | 系统先存为草稿，到达时间后自动发布 |
| 需要复用统一文案发布通知 | 选择模板并传入变量 | 系统按模板渲染各渠道内容后发布 |
| 发布前需要核对文案与渠道适配 | 发起发布预览 | 系统返回各渠道渲染内容与预估收件人数 |
| 需要按事件配置自动通知 | 创建规则（触发类型 + 模板 + 渠道） | 规则启用后，对应事件发生时自动投递通知 |
| 业务事件需要自动通知 | 系统在租户开通/用户激活等节点自动触发已配置规则 | 通知服务按规则匹配模板、接收收件人并投递 |
| 需要核对触达效果 | 查看通知投递结果 | 系统展示各收件人各渠道的投递状态 |
| 需要核对整体触达效果 | 查看通知请求投递统计 | 系统展示按渠道分组的成功/失败/跳过/超时计数 |
| 需要确认渠道配置可用 | 对渠道配置发送测试通知 | 系统按配置发送测试通知并反馈结果 |

**普通业务用户**

| 触发场景 | 用户动作 | 系统结果 |
| --- | --- | --- |
| 用户需要查看与自己相关的信息 | 进入“我的通知” | 系统展示该用户可见的通知列表与未读数 |
| 用户已阅读一条通知 | 执行已读操作 | 系统记录该通知对该用户的已读结果 |
| 某条通知要求明确确认 | 执行确认操作 | 系统记录确认结果，供后续追踪 |
| 用户希望调整接收方式 | 设置通知偏好 | 系统按偏好过滤后续投递渠道 |

### 约束与边界

- 主归属模块为“通知中心”，次级关联模块为“安全与审计”。
- 通知服务不依赖系统管理服务：集成方向为单向 `系统管理 -> 通知中心`（首版 HTTP，契约传输层无关，预留消息队列切换），收件人由调用方解析后传入。
- 站内通知（STATION 渠道）在发布时按收件人每渠道预生成投递记录（回执），支持已读、确认与结果追踪；邮件/短信/Push/Webhook 等渠道的投递结果同样以回执记录，仅 STATION 渠道进入“我的通知”。
- 事务性发送（如多因素验证 OTP）通过指定渠道同步发送，不生成站内信，敏感内容脱敏存储，不落明文。
- 渠道提供商凭证加密存储，不明文落库。
- 通知分为系统通知与租户通知：系统通知由平台发出（`tenantId` 为空），收件人可跨租户，使用平台级渠道配置，默认不受用户偏好约束；租户通知由租户发出，收件人在租户边界内，使用租户级渠道配置（缺失回退平台级），默认受用户偏好约束。
- 通知审批流和营销型活动运营不在通知中心范围。
- 通知按“目标范围”组织，目标范围到收件人的解析由调用方负责。

### 分阶段说明

| 阶段 | 范围 |
| --- | --- |
| 阶段 1 基础闭环 | 发布编排（立即/定时）、按目标范围投放、我的通知、已读、确认、结果追踪、站内与邮件渠道、平台级渠道配置、`/notices/send` 事务性发送（OTP 等） |
| 阶段 2 模板中心 | 通知模板管理、按渠道配置、变量渲染、发布接入模板、发布预览 |
| 阶段 3 规则自动化 | 通知规则管理、事件触发、触发接收口与系统管理接入 |
| 阶段 4 偏好与多渠道 | 用户通知偏好、短信/Push/Webhook/钉钉等渠道扩展、租户级渠道配置（多提供商）、请求级投递统计 |

## 6. 跨模块公共能力

| 公共能力 | 主归属模块 | 次级关联模块 | 需求说明 |
| --- | --- | --- | --- |
| 身份与访问资格 | 认证与授权 | 系统管理、安全与审计 | 不同角色登录后应进入与其身份匹配的访问范围 |
| 租户边界 | 租户管理 | 认证与授权、系统管理、通知中心、安全与审计 | 所有与租户有关的业务行为都应在明确租户边界下发生 |
| 套餐与运营边界 | 租户套餐 | 租户管理、配置管理 | 租户可用能力、有效期和基础规模边界需要有统一业务载体 |
| 菜单与功能可见范围 | 系统管理 | 认证与授权 | 用户看到的入口和可访问范围必须一致 |
| 通知触达与反馈 | 通知中心 | 安全与审计 | 发布、模板、规则、查看、已读、确认、结果追踪形成完整闭环 |
| 关键动作留痕 | 安全与审计 | 认证与授权、租户管理、系统管理、通知中心 | 登录、开通、授权、发布等关键动作都应可追踪 |
| 配置与能力边界 | 配置管理 | 全部模块 | 平台级配置和能力边界要能约束其他模块的可用范围 |

## 7. 当前版本不做的内容

- 不交付公共入口与工作区前端页面。
- 不建设独立任务中心或通用消息运营台。
- 不建设客户体系和临时登录令牌能力。
- 不交付完整的外部身份接入能力。
- 不交付多因素验证能力。
- 不建设通知审批流和营销型活动运营。
- 不建设复杂商业计费、续费和账务体系。
- 不建设复杂数据范围治理、复杂组织治理和复杂安全运营中心。
- 不引入新的示例业务域。

## 9. 文档冲突点

| 冲突主题 | 冲突表述 | 当前采用口径 |
| --- | --- | --- |
| 多租户默认状态 | 既有需求文档出现“默认关闭”表述，其他文档和当前仓库状态出现“默认启用，可按需关闭隔离”表述 | 以当前仓库真实状态为准，按“当前已具备租户主链路，是否关闭隔离属于运行边界控制”处理 |
| 公共入口与工作区 | 既有需求文档保留了公共入口和工作区需求，当前开发计划明确不做前端页面 | 从正文模块中移出，仅保留为当前版本不做内容 |
| 消息与任务 | 既有需求文档把“消息与任务”列为核心范围，当前仓库没有独立任务中心 | 仅保留为“服务核心链路的后台处理需求”，不扩展为独立产品模块 |
| 客户与临时登录令牌 | 既有需求文档写入了后续方向，参考文档同时说明该能力缺少当前仓库归属 | 不视为既定需求，列入待确认项 |
| 租户状态范围 | 既有需求文档对“开通、待激活、启用、停用、到期、禁用”关系表述不一致，且未明确初始化发生在哪个阶段 | 采用“创建即开通但初始为待激活；首个管理员激活并完成初始化后进入启用；到期自动停用；停用允许受限访问但禁止写；禁用完全不可用”的口径 |
