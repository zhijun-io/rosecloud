# RoseCloud 需求说明

## 1. 文档定位

本文档从需求角度整理 `rosecloud`，用于统一项目定位、能力边界、角色场景和当前确认版需求基线。

当前阶段不讨论：

- 详细模块实现
- 具体代码结构
- 具体框架接入细节
- 复杂架构模式落地方案

## 2. 项目定位

`rosecloud` 定位为一个可长期演进的企业级后台与微服务基础平台。

它的目标不是做一个只适合演示的脚手架，而是：

- 能支撑真实业务启动
- 能兼顾单体开发体验与微服务演进空间
- 能在不推翻重来的前提下逐步增强能力
- 能被中大型业务团队长期维护

## 3. 核心建设目标

### 3.1 运行形态目标

- 支持单体模式运行
- 支持微服务模式运行
- 单体模式用于本地开发、联调和中小规模部署
- 微服务模式用于独立部署、扩缩容和后续拆分演进

### 3.2 架构目标

- 面向接口编程
- 强调分层、解耦、可测试、可演进
- 不采用复杂 DDD 四层强约束
- 不为了未来可能用到的场景提前做过重抽象

### 3.3 演进目标

- 当前阶段服务间通信使用 OpenFeign
- 后续具备平滑替换为 Dubbo 的能力
- 当前阶段先确定需求边界，再逐步落地实现

## 4. 当前确认版需求基线

本节内容基于当前对 `rosecloud` 的连续需求梳理结果，作为现阶段最权威的需求基线。

### 4.1 产品定位

- `rosecloud` 是企业后台 + SaaS 平台底座，不是单一业务系统
- 第一版不是空框架，而是可完整演示的平台产品
- 第一版带一个轻量示例业务域，用来证明平台可以承载真实用户闭环；前端门户与完整工作区作为后续交付项，不纳入 P0 完整范围

### 4.2 第一版核心能力包

第一版采用“标准平台包”，至少包括：

- 认证与授权
- 系统管理
- 租户管理
- 通知中心
- 消息与任务
- 多数据源基础能力
- 网关与统一入口
- 单体 / 微服务双模式

### 4.3 第一版核心闭环

第一版的核心不是“模块堆叠”，而是下面这条平台闭环真实成立：

`租户申请或平台直开 -> 审核 -> 标准开通 -> 初始化租户与管理员 -> 租户管理员运营租户 -> 普通用户登录使用 -> 公告/通知触达 -> 已读/确认反馈 -> 平台侧审计、任务、监控可追踪`

> “可完整演示”指本闭环端到端可走通；其中“监控可追踪”在 P0 指健康检查 + 日志 + 审计可查，完整监控与告警为 P2。

### 4.4 角色体系

第一版至少包含以下角色：

- 平台管理员
- 租户管理员
- 普通业务用户
- 安全管理员

其中：

- 平台管理员后台定位为“综合运营中心”
- 租户管理员后台定位为“租户运营中心”
- 普通业务用户进入租户侧入口，拥有基础工作台
- 安全管理员第一版可由平台管理员兼任

说明：

- 上述角色中，平台管理员、租户管理员、普通业务用户、安全管理员属于产品内正式业务角色
- 运维人员、开发人员属于平台建设与交付协作角色，不作为第一版默认内置业务角色要求

### 4.5 前端产品结构

前端结构作为后续交付项，需支持：

- 公共入口区
- 平台管理员工作区
- 租户管理员 / 普通业务用户工作区

边界要求：

- 公共入口区与后台工作区属于同一产品体系，但明显分区
- 平台侧和租户侧不是两套独立产品
- 普通业务用户与租户管理员共用租户侧入口，但菜单和权限不同
- 前端必须明确体现当前租户上下文

### 4.6 公共入口区

公共入口区定位为“平台官网式入口”，但不是重营销站。该能力放在后续交付阶段。

至少承担以下职责：

- 平台定位说明
- 核心能力说明
- 适用场景说明
- 登录入口
- 租户申请入口

登录入口采用混合模式：

- 支持统一登录入口
- 也支持平台侧 / 租户侧直达登录页

登录租户解析：用户名 / 邮箱在租户内唯一（非全局唯一），各租户可各自存在同名用户；普通用户登录需携带租户标识（租户编码），平台管理员登录无需租户标识；统一入口需填 / 选租户，租户侧直达入口自带租户上下文。

### 4.7 租户申请与开通

第一版租户开通有两条正式路径：

- 平台管理员直开
- 外部租户申请后审核开通

租户申请路径要求：

- 采用审核型，而不是纯登记或复杂审批流
- 审核通过后直接进入标准开通流程
- 申请人与首个租户管理员可以相同，也可以不同
- 申请信息采用标准申请型，至少覆盖租户名称、联系人、联系方式、申请说明、使用规模和首个管理员信息
- 审核结果通知采用双通道：平台内留痕 + 外部至少一种触达方式；首个管理员激活闭环作为后续增强项

### 4.8 租户开通后的标准初始化

租户开通后至少自动完成以下动作：

- 创建租户记录
- 创建首个租户管理员账号
- 初始化默认角色骨架
- 初始化默认菜单 / 权限骨架
- 初始化基础配置骨架
- 触发基础通知 / 任务初始化动作；首个管理员激活流程作为后续增强项

### 4.9 角色、菜单与权限骨架

租户侧默认角色骨架至少包括（平台管理员、安全管理员为平台侧内置角色，见 §4.4 / §6.1）：

- 租户管理员
- 普通业务用户

默认菜单 / 权限骨架至少包括：

- 租户管理员：用户管理、角色 / 菜单权限、基础配置、通知查看 / 局部管理、任务查看 / 有限操作、日志 / 审计基础查看
- 普通业务用户：个人首页或工作台、个人通知、被授权基础页面

租户管理员权限边界：

- 可在本租户范围内管理用户、角色、菜单授权关系
- 不能突破平台定义的菜单体系和能力边界

菜单配置要求：

- 每个租户的菜单不要求完全一致
- 平台应提供“平台定义能力边界 + 租户侧按授权范围配置菜单”的模式
- 租户管理员可基于平台下发的菜单与权限骨架，配置本租户可见菜单、启用状态、展示层级和角色授权关系
- 租户侧可做菜单裁剪与组织，不可凭空创建超出平台能力边界的功能点

权限模型中的“权限点与菜单分离”作为后续增强项处理。P0 仅保证菜单与基础授权闭环可用；更细的按钮 / 接口级权限点拆分、租户按能力边界配置权限点，放到后续需求中。数据权限（按部门 / 组织限定可见数据）仍为 P1。

### 4.10 平台管理员跨租户支持

平台管理员可以受控进入租户视角，用于支持、排障、审核和治理。

边界要求：

- 允许受控代操作
- 不等同于无限超级用户
- 必须单独增强审计，至少记录：谁进入、进入哪个租户、做了什么、何时进入、何时退出

代操作机制与边界：采用 impersonation（以租户管理员身份进入，会话标记为代操作）；只读优先，写操作需二次确认；禁止删除租户、修改套餐、修改租户管理员凭据等高危操作；审计会话级标记代操作人与被代租户，与正常租户会话区分。该能力放到后续阶段。

### 4.11 租户模型、状态与套餐

多租户不是纯预留能力，而是第一版主路径之一。

要求：

- 租户能力可配置启用，默认关闭
- 启用后第一阶段优先支持 `COLUMN`
- `SCHEMA`、`DATASOURCE` 作为后续演进方向

说明：

- “默认关闭”指部署默认开关值，不代表第一版不做多租户能力
- 第一版仍需真实支持租户主链路，并在开启租户能力后能够完整跑通申请、开通、初始化、登录、授权、通知和审计闭环

租户状态第一版至少包括：

- 待开通
- 启用
- 停用 / 冻结
- 过期

状态行为定义：

- `过期`：商业状态，有限降级（非硬阻断、非纯提示）；可登录、可读、禁写，前端显著提示；触发于套餐到期，续费后恢复为 `启用`
- `停用 / 冻结`：管理限制，管理员手动触发，阻断登录

套餐配额超限行为：超出上限（如用户数）时阻断新增，存量不受影响，并提示预警。

套餐 / 配额要求：

- 第一版是轻量生效，不是纯字段占位
- 先作用于租户基础能力，例如可用性、用户数上限、基础功能开关
- 不做第一版完整商业化计费体系
- 配额预警：接近上限时轻量提示（不做计费）

### 4.12 认证与安全基线

默认认证主路径：

- 账号密码登录
- JWT
- 刷新机制

第一版即需明确占位的增强能力：

- OAuth2
- MFA

要求：

- 默认不启用
- 但在产品边界和配置开关上必须明确存在
- 登录续期、登录审计和安全审计进入正式需求范围
- 会话管理 P0 范围：在线会话查询、强制下线、登出按 jti 吊销令牌；更完整的会话治理（单用户会话数上限、改密后失效旧令牌）为 P1（见 §5.2）
- 登录审计记录来源 IP 与 User-Agent（浏览器 / 操作系统 / 设备）

后续安全增强项包括：

- 密码策略可配置（最小长度、大小写 / 数字 / 特殊字符、过期天数、禁止重用）
- 失败登录锁定（连续失败超阈值锁定账号，并可通知）
- 用户激活与密码重置走限时令牌链接
- 临时登录令牌（客户自助申请，短 TTL + 可吊销 + scope 受限）为 P1（见 §4.18）

### 4.13 通知中心

通知中心第一版定位为“平台服务型”，不是单纯中间件适配。

角色边界：

- 平台管理员：管理通道、全局策略、平台级模板与治理
- 租户管理员：管理本租户通知使用、局部模板 / 开关、查看记录
- 普通业务用户：接收和消费通知

第一版触达主路径：

- 站内通知
- 邮件

辅助通道：

- 短信

### 4.14 租户开通异步化

租户开通第一版不建独立任务中心，直接使用进程内异步执行首个管理员创建和租户启用。

约束：

- 只覆盖租户开通这一条主链路
- 不提供任务列表、重试、取消、调度配置等独立能力
- 后续若出现新的明确异步编排需求，再单独评估是否引入任务编排模型

### 4.15 系统管理

系统管理第一版采用“可运营平台型”深度，至少包括：

- 用户管理
- 角色管理
- 菜单管理
- 权限点管理
- 参数配置
- 安全设置（密码策略、失败锁定阈值、令牌 TTL 集中配置）
- 字典与基础平台数据
- 部门与组织基础信息
- 登录日志与操作日志
- 在线会话（在线用户）管理：查询在线用户、强制下线（见 §4.12 会话管理）

### 4.16 示例业务域

第一版示例业务域选型为“公告 / 信息发布类”。

业务边界：

- 平台管理员可发布平台级信息，或面向指定租户发布
- 租户管理员可发布本租户内部信息
- 普通业务用户负责消费信息

能力要求：

- 投放粒度支持全局 / 租户 / 角色
- 不做第一版指定用户级精细投放
- 反馈模式支持“已读 + 确认”
- 不走发布审核
- 支持即时发布 + 定时发布（定时发布指发布动作在指定时间触发）
- 支持生效时间 / 失效时间（指内容发布后的可见时间窗，与发布触发时间相互独立）

普通业务用户入口：

- 工作台中的摘要与待处理入口
- 独立“我的消息 / 公告中心”列表页

### 4.17 普通业务用户闭环

普通业务用户不是静态账号对象，而必须真实进入闭环。

第一版至少具备：

- 被创建
- 可登录
- 可被授权
- 可访问被授权页面
- 可接收通知
- 可查看公告 / 信息
- 可完成已读 / 确认

普通用户首页定位为“基础工作台”，至少包含：

- 与自己相关的事项或结果
- 最近通知
- 常用入口
- 基础状态提示

### 4.18 客户与临时令牌登录

借鉴 ThingsBoard 的 Customer 模型，租户下引入“客户”（Customer）：

- 客户归属租户，租户管理员可管理本租户客户
- 普通业务用户可归属某客户（客户用户），不新增独立角色
- 客户用户可申请临时登录令牌：短 TTL、可吊销、scope 受限（仅客户可访问范围）
- 临时令牌默认关闭，开关门控（如 `rosecloud.customer.temp-token.enabled`）
- 客户与临时令牌均为 P1（默认关闭）；客户实体可按需提前到 P0

## 5. 平台能力范围与分阶段边界

**术语与占位级别**：本文档中“第一版 / 第一阶段 / P0”同义。占位能力分三级——L0 仅配置开关；L1 开关 + 端点签名（前端可识别能力存在但未启用）；L2 完整实现，开关开启即用。默认规则：P0 占位项到 L1，P1 补到 L2；多租户、基础缓存 P0 即 L2；AI 模块、Excel 不在 P0。

### 5.1 P0：第一阶段必须进入范围

第一阶段必须纳入的能力包括：

- 认证与授权
- 系统管理
- 租户管理
- 通知中心
- 消息与任务
- 多数据源基础能力
- 网关与统一入口
- 单体 / 微服务双模式
- 轻量示例业务域

### 5.2 P1：第二阶段应补齐

第二阶段重点补齐：

- 前端公共入口区与后台工作区
- 登录安全增强：密码策略、失败登录锁定、用户激活 / 密码重置强化、临时登录令牌
- 租户申请审核 / 开通 / 首个管理员激活闭环
- 权限点与菜单分离、按钮 / 接口级权限点治理
- OAuth2 接入能力：第三方 / 社交登录（客户端模式），租户级提供商配置，首次登录可选自动建用户；不含完整 Authorization Server
- MFA 能力：TOTP 为主，短信 / 邮件 / 备份码按需，pre-auth 令牌流程 + 限频锁定；不含全量场景
- 更细粒度的数据权限：数据范围视图（按角色 / 组织限定可见数据）
- 更完整的会话治理：单用户会话数上限、改密后失效旧令牌
- 租户生命周期、套餐与配额完善：实体配额（用户 / 资源数）+ 限流 + 数据 TTL + 配额预警
- 文件存储、Excel、分布式锁、业务流水号等平台支撑能力补齐到 L2（缓存 P0 已可用）
- 更完整的测试、CI/CD 与交付质量能力
- 客户（Customer）实体与临时令牌登录（客户自助申请，短 TTL / 可吊销 / scope 受限，默认关闭，见 §4.18）

### 5.3 P2：后续可扩展能力

后续按真实需要再推进：

- Dubbo 适配
- Kafka / ActiveMQ 适配
- 更强的租户隔离模式
- 更复杂的灰度、限流、熔断与治理能力
- AI 模块正式接入
- 更完整的平台化脚手架和自动生成能力
- 通知规则自动化（触发器 → 目标 → 模板）
- 看板与组件库（如需可视化）
- 告警 / 事件闭环（如需运维告警）
- 实体 / 配置版本控制

### 5.4 默认关闭但必须预留的能力

以下能力不建议第一阶段默认开启，但必须在需求边界中明确占位：

- OAuth2
- MFA
- AI 模块
- 更重的安全治理能力
- API Key 认证
- 移动端登录（QR 扫码 / 移动应用）

其中多租户为 P0 实现（仅默认关闭，见 §4.11），不属纯预留；OAuth2 / MFA 在 P0 到 L1（开关 + 端点签名），P1 补到 L2；AI 模块 P0 不实现。

### 5.5 明确不在第一阶段做满的能力

以下能力不纳入第一阶段完整实现：

- 完整 OAuth2 Authorization Server 体系
- 全量 MFA 场景覆盖
- `SCHEMA` 与 `DATASOURCE` 的完整生产实现
- Dubbo 主通信路线
- Kafka / ActiveMQ 双套消息实现
- 复杂规则引擎
- 复杂工作流引擎
- 深度 AI 平台化

## 6. 角色与业务场景

### 6.1 核心角色

第一阶段至少包含以下角色：

- 平台管理员
- 租户管理员
- 普通业务用户
- 安全管理员

角色定位：

- 平台管理员：综合运营中心，负责平台治理、租户开通、平台配置、安全策略、平台总览
- 租户管理员：租户运营中心，负责本租户用户、角色、配置、通知和任务的有限管理
- 普通业务用户：租户侧真实使用者，拥有基础工作台和消息 / 公告消费能力
- 安全管理员：第一阶段可由平台管理员兼任；负责安全设置域（密码策略、失败锁定阈值、令牌 TTL 等集中配置，见 §4.15），无业务数据管理权限

支撑角色说明：

- 运维人员、开发人员属于平台建设与交付支撑角色，不作为第一版默认产品内置业务角色

### 6.2 第一阶段核心业务场景

第一阶段核心场景包括：

- 平台初始化与环境准备
- 用户登录与访问控制
- 平台管理员开通与管理租户
- 租户管理员管理本租户用户与权限
- 通知发送与异步处理
- 定时任务与后台处理
- 平台运行监控与审计

### 6.3 公共入口与工作区场景

前端产品至少包含：

- 公共入口区：平台介绍、能力说明、登录入口、租户申请入口
- 平台管理员工作区
- 租户管理员 / 普通用户工作区

登录采用混合模式：

- 支持统一登录入口
- 支持平台侧 / 租户侧直达登录页

### 6.4 示例业务场景

第一阶段示例业务域采用“公告 / 信息发布类”。

核心场景包括：

- 平台管理员发布平台级或指定租户信息
- 租户管理员发布本租户内部信息
- 普通业务用户查看、已读或确认信息
- 平台与租户侧追踪触达和确认结果

## 7. 核心用户用例

本节仅保留 P0 关键用户用例，用于支撑原型、接口、权限和验收设计。

| 用例名称 | 参与者 | 目标 | 主流程 | 结果 |
|---|---|---|---|---|
| 提交租户申请 | 外部申请方 | 提交入驻申请，进入审核流程 | 在公共入口填写租户、联系人和首个管理员信息后提交申请 | 系统生成可跟踪的申请记录 |
| 审核并开通租户 | 平台管理员 | 完成审核并正式开通租户 | 审核申请后创建租户、首个管理员，并初始化角色、菜单、权限和基础配置；首个管理员激活流程放到后续阶段 | 租户可管理，管理员信息已初始化，过程可审计 |
| 首次登录并初始化租户 | 租户管理员 | 完成租户启用后的基础检查和初始化确认 | 登录租户侧入口后检查基础配置、角色菜单骨架和通知设置 | 租户进入可运营状态 |
| 配置本租户菜单与角色授权 | 租户管理员 | 形成符合本租户需要的菜单和权限结构 | 在平台能力边界内裁剪菜单、调整层级，并给角色和用户授权 | 不同租户、不同角色可看到不同菜单 |
| 创建并授权普通用户 | 租户管理员 | 让普通用户进入真实使用链路 | 创建用户账号并分配角色、菜单权限和组织信息 | 普通用户可登录并访问被授权功能 |
| 登录并使用工作台 | 普通业务用户 | 进入租户侧工作区完成基础使用 | 登录后查看工作台、通知、常用入口和被授权页面 | 普通用户形成真实使用闭环 |
| 发布并触达公告 | 平台管理员、租户管理员 | 向目标用户发布信息并完成触达 | 创建公告，选择投放范围和时间后发布，通过站内或邮件触达用户 | 用户可在工作台或公告中心看到公告 |
| 查看并确认公告 | 普通业务用户 | 完成公告阅读和确认 | 从工作台或公告中心进入详情后执行已读或确认 | 系统保存公告消费结果并可追踪 |
| 受控进入租户视角处理问题 | 平台管理员 | 在合规前提下协助租户排障和治理 | 进入目标租户视角后查看配置、排查问题或执行有限代操作 | 支持过程全量留痕，可审计 |
| 处理失败任务或通知 | 平台管理员、租户管理员 | 对失败的异步任务或通知进行处理 | 查看失败原因后执行重试或人工处理，并跟踪结果 | 失败记录具备可追踪、可重试的闭环 |

## 8. 平台支撑能力要求

### 8.1 网关与接口治理

需要支持：

- API Gateway 统一接入
- 统一路由转发
- 统一认证入口
- 公开接口与受保护接口分层管理
- 统一上下文透传
- 请求链路标识透传

需要预留的平台治理能力包括：

- 接口限流
- 幂等控制
- 接口审计
- 数据权限扩展位
- 黑白名单与安全开关

### 8.2 多数据源能力

第一版多数据源定位为“平台基础型”。

至少真实存在两类数据源场景：

- 平台元数据
- 业务数据

并为后续租户数据路由预留扩展位。

边界说明：

- 第一版要求具备多数据源抽象、配置和路由扩展位
- 第一版允许开发环境或演示环境采用单实例、逻辑分库或同实例双库方式启动
- 第一版不强制所有场景都实现物理独立数据源部署
- 当进入更高隔离要求的环境时，再启用独立数据源或租户专属数据源策略

### 8.3 平台基础设施能力

需求上需要占位但不要求第一阶段全部做满的能力（P0 到 L1：开关 + 端点签名，P1 补到 L2）：

- 文件存储（P0 L1，P1 L2）
- 业务流水号或序列号（P0 L1，P1 L2）
- 缓存（P0 L2，基础缓存可用）
- 分布式锁（P0 L1，P1 L2）
- 可插拔规则扩展能力（P0 L1 扩展点；复杂规则引擎见 §5.5）

不在 P0 的项：Excel 导入导出（P1）、AI 模块（P2）。

### 8.4 配置与功能开关

需要支持：

- 共享基础设施配置
- 服务级配置
- 本地默认配置
- 环境变量覆盖
- 不同环境配置隔离

需要支持的功能开关至少包括：

- 租户能力
- OAuth2
- MFA
- AI 模块
- 后续安全增强或集成功能

能力开关应集中暴露 API，供前端查询各能力启用状态。

约束：

- 所有增强能力默认按需开启
- 第一阶段优先支持“默认关闭 + 可配置开启”

## 9. 当前阶段结论

现阶段对 `rosecloud` 的需求已经可以收敛为：

- 第一版做平台底座，不做空平台，也不做重业务系统
- 第一版核心价值是跑通“租户申请 / 开通 / 初始化 / 租户运营 / 普通用户消费 / 通知任务 / 审计追踪”闭环
- 第一版以平台能力为主，同时带一个轻量示例业务域
- 所有增强能力按“可选、按需、分阶段”处理

## 10. 需求侧的关键原则

- 不默认复刻当前项目的模块结构
- 不默认照搬 Dubbo、Sa-Token、DDD 等现有设计
- 只吸收当前项目中真正有参考价值的工程经验
- 第一阶段优先做减法，而不是做满
- 所有增强能力都应考虑“默认关闭、按需开启”
- 需求文档要把平台型基础能力先占位，再决定哪些进入第一阶段
